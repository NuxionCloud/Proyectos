1 · Overview

Web de Bocatitos (Av. Sánchez Pizjuán 6, Sevilla). Tres pilares diferenciados de negocio: desayuno (solo consumo en local), tapas y street food (ambos con domicilio).

Estética agresiva oscura inspirada en streetfoodburger.com pero resolviendo el "problema de las 3 categorías" (la referencia solo vende burgers, Bocatitos comunica las tres líneas desde el hero).

3 · Pre-requisitos

  • Node.js 18.17+ (probado en Node 24.15)
  • npm 9+ (pnpm/yarn también funcionan, ajustar comandos)
  • Acceso al repo de modelos en proyectos/bocatitos-redesign/modelo-11-cinematic-editorial/ (fuente del seed de contenido)
  • Editor con soporte Astro (VS Code + extensión astro-build.astro-vscode)
  • Navegador moderno para validar (Chrome / Edge / Firefox)

4 · Reglas de oro

1. La carta NO lleva precios. Decisión explícita del cliente: "Sin precios en la web — pregúntanos por WhatsApp". El schema Zod no acepta el campo precio. Si en el futuro cambia, se añade como opcional.

2. La carta vive en i18n.js del modelo 11. Es la fuente validada por el cliente. El seeder seed-content.mjs regenera los 23 .md desde ahí. Nunca inventar nombres ni ingredientes.

3. Desayuno = solo local. disponibleDomicilio: false en el frontmatter. La página /pedidos filtra los desayunos por esto.

4. Sin backend. Pedidos y reservas se envían por WhatsApp con mensaje pre-rellenado. Si se añade backend en v2, mantener WhatsApp como fallback.

5. Cada categoría tiene su sub-paleta. Verde (desayuno), terracota (tapas), amarillo (street food). El hilo de marca es negro + rojo + amarillo (chrome). Hover y borders de las cards usan la sub-paleta.

6. Performance primero. JS bundle inicial < 10 KB gzipped, fuentes preconnect, imágenes loading="lazy" excepto hero (fetchpriority="high"), Google Maps lazy al click, todas las cards con width/height para evitar CLS.

5 · Procedimiento paso a paso (setup desde cero)

5.1 · Clonar / colocar el proyecto

cd "d:\Users\rafar\OneDrive\Escritorio\Claude\proyectos\bocatitos-redesign\_app"

5.2 · Instalar dependencias

npm install

Instala Astro 4.16, Tailwind 3.4, MDX, GSAP, TypeScript, Prettier. ~60 segundos.

5.3 · Generar contenido (si la carpeta content/ está vacía)

npm run seed

Lee ../modelo-11-cinematic-editorial/assets/i18n.js y genera:

  • 17 archivos en src/content/street-food/
  • 3 archivos en src/content/desayunos/
  • 3 archivos en src/content/tapas/
  • public/i18n/dishes.json (índice multi-idioma para el switcher)

5.4 · Arrancar dev server

npm run dev

Sirve en http://localhost:4321. Hot reload activado.

5.5 · Build de producción

npm run build

Ejecuta prebuild (genera sitemap), luego astro check (type-check) y astro build (genera dist/ con HTML estático).

5.6 · Preview del build

npm run preview

Sirve dist/ en local para validar antes de subir.

6 · Daily workflow

  1. npm run dev · arranca dev server
  2. Edito componentes, páginas, tokens.css, contenido .md
  3. Hot reload muestra cambios en milisegundos
  4. Validar visual en localhost:4321 — siempre antes de subir
  5. Ctrl+C para parar el server
  6. npm run build · genera dist/
  7. npm run preview · validar el build final
  8. Subir dist/ al host (FTP a Hostinger, push a GitHub si autodeploy, etc.)

7 · Editar la carta

7.1 · Añadir un plato nuevo

Crear archivo en src/content/<categoria>/NN-slug.md con frontmatter:

---
nombre:
  es: "TITO NUEVO"
  en: "TITO NUEVO"
  fr: "TITO NUEVO"
  it: "TITO NUEVO"
  de: "TITO NUEVO"
base: "brioche"         # brioche | ciabatta | bun | plato
subcategoria: "smash"   # smash | bocata | hotdog | desayuno | tapa
categoria: "street-food"
imagen: "https://..."
tag: "signature"        # opcional: signature | spicy | veggie
destacado: true         # opcional, sale en featured de home
disponibleDomicilio: true
orden: 18               # número para ordenar
ingredientes:
  es:
    - "Ingrediente 1"
    - "Ingrediente 2"
  en: [...]
  fr: [...]
  it: [...]
  de: [...]
---

7.2 · Ocultar un plato (sin borrarlo)

Añadir disponible: false al frontmatter. Reaparece al ponerlo true.

7.3 · Regenerar todo desde el modelo 11

Si la carta del modelo 11 cambia (validada por el cliente):

npm run seed

Borra y recrea los 23 .md + el JSON i18n. Idempotente.

7.4 · Editar datos del local

Editar src/content/settings/local.json (dirección, horarios, WhatsApp, IG). Se refleja en footer, schema.org y página de reservas.

8 · Deploy

8.1 · Hostinger (recomendado, sigue la línea Nuxion)

  1. Generar build: npm run build
  2. Subir el contenido de dist/ a la raíz del dominio en hPanel (FTP o file manager)
  3. Verificar que sitemap.xml y robots.txt están en la raíz
  4. Si el dominio es bocatitos.es: configurar en hPanel → Dominios

8.2 · Auto-deploy con GitHub Actions (opcional)

Idéntico al setup de nuxion.cloud: push a main → workflow que hace npm install + npm run build y sube por FTP.

8.3 · Vercel / Netlify (alternativa)

Detección automática de Astro. Build command: npm run build. Output: dist/. Cero config.

8.4 · GitHub Pages

Añadir base: '/<repo>/' a astro.config.mjs si la web cuelga de subpath.

9 · Troubleshooting

SíntomaCausaFix
Cannot read properties of undefined (reading 'reduce') al build Bug del plugin @astrojs/sitemap con esta combo de versiones Sitemap se genera con scripts/gen-sitemap.mjs (ya en prebuild). Si vuelve a fallar, dejar comentado sitemap() en astro.config.mjs.
Rollup failed to resolve import "@i18n/ui" Falta alias en Vite Ya está añadido en astro.config.mjs → vite.resolve.alias y en tsconfig.json → paths. Si añades otro alias, replícalo en ambos sitios.
El switcher de idioma no cambia los ingredientes Las cards no tienen data-dish-slug, o el fetch a /i18n/dishes.json falla Verificar que las cards tienen data-dish-slug={it.slug} y que el npm run seed ha generado public/i18n/dishes.json.
Las fuentes tardan en aparecer (FOIT) Sin preload El BaseLayout ya hace preconnect + preload de Google Fonts. Si añades otra fuente, replicar el patrón.
El mapa de Google no se ve El iframe es lazy: requiere primer click Es intencional (ahorra ~700 KB). Al hacer click sobre la imagen el iframe se carga.
El TopBar marquee no se anima prefers-reduced-motion: reduce activo en el sistema Es intencional (accesibilidad). Probar con preferencia "no preference" o en navegación privada.
Unexpected token ';' al correr el seed Cambió la estructura de window.T en i18n.js del modelo 11 El parser empareja llaves; si cambia la sintaxis (p. ej. de window.T = {} a export const T = {}), ajustar loadI18n() en seed-content.mjs.

10 · Decisiones de diseño tomadas

10.1 · Sin GSAP en el hero

El brief original pedía GSAP. El efecto descrito (stagger eyebrow → títulos → cards con fade-up) se resuelve perfectamente con CSS animation-delay. Decisión: CSS-only para el hero. GSAP queda como dependencia para v1.1 si se introducen efectos ScrollTrigger en categorías expandidas (parallax).

10.2 · Sin CMS visual (Decap fuera)

El brief proponía Decap CMS. Decisión: contenido en Markdown directo. La carta cambia muy poco y la edita el equipo técnico. Si en v2 quieren editor visual, se monta Decap por encima sin cambiar el contenido.

10.3 · i18n light (cliente) en v1, routing completo en v1.1

Implementar i18n con páginas duplicadas para 5 idiomas (Astro routing) requiere refactor masivo de todos los componentes. Decisión: en v1 los ingredientes y la "base" de la carta cambian en cliente (los 5 idiomas ya están en los .md). El UI estructural queda en ES y se traduce en v1.1.

10.4 · Sin precios en la web

Decisión explícita del cliente. El schema Zod ni siquiera contempla el campo. Los precios se consultan por WhatsApp.

10.5 · Reservas y pedidos vía WhatsApp

No CoverManager, no TheFork, no Glovo. Local pequeño → el flujo es siempre por WhatsApp con mensaje pre-rellenado. Los formularios de /pedidos y /reservas componen el mensaje y abren wa.me.

10.6 · Color de tapas pendiente de confirmación

El cliente definió colores para desayuno (verde + amarillo) y street food (amarillo + negro). Para tapas no especificó. Propuesta actual: terracota #C8553D (cerámica sevillana, contrasta con verde y amarillo). Pendiente validar con el cliente.

12 · Changelog · lecciones aprendidas

v1.0 · 2026-05-21

  • Lección: el cliente NO quiere precios en la web. Hubo que rehacer el schema Zod a mitad de implementación. Documentar restricciones de negocio antes de modelar.
  • Lección: nunca inventar contenido. La carta vive en el modelo 11 (i18n.js). Crear un seeder es la forma correcta de mantenerlo sincronizado.
  • Lección: @astrojs/sitemap v3.2.1 tiene un bug con builds de pocas páginas. Mejor un script propio: 30 líneas, cero dependencia, total control.
  • Lección: sub-paletas por categoría enriquecen la marca sin romper la consistencia. Cada bloque expandido respira su color, el chrome mantiene la línea (negro + rojo + amarillo).
  • Lección: i18n full-routing con 5 idiomas multiplica el trabajo por 5. Empezar por i18n cliente-side para lo más visible (ingredientes) entrega valor sin bloquear v1.
  • Lección: el aliases en Vite (resolve.alias) DEBE coincidir con los del tsconfig (paths). Si solo está en uno, build falla pero IDE acepta.

v1.2 · 2026-05-27

  • Lección: el handoff entre un overlay animado y un elemento "fijo" del header (en este caso el logo pequeño) necesita medir getBoundingClientRect() en JS. Cualquier calc() con vw/vh falla cuando hay capas variables encima (TopBar marquee de altura diferente a otros sitios). La fórmula manual nunca acierta en todos los viewports.
  • Lección: esperar document.fonts.ready antes de medir geometría. Si la fuente recarga después, los boundingClientRect cambian de tamaño/posición y la animación cae mal.
